# Seolin SafeCheck PRD

## 1. 문서 목적

이 문서는 **Seolin SafeCheck**의 제품 요구사항과 MVP 범위를 정의합니다. Seolin SafeCheck는 서린태권도 등원 차량 기사님이 원생의 차량 탑승 여부를 쉽게 기록하고, 해당 정보를 관리자 대시보드, 데이터베이스 기록, 엑셀 내보내기, 학부모 문자 알림으로 연결하는 스마트 차량 출결 관리 시스템입니다.

본 PRD는 향후 Codex 기반 개발 작업에서 제품 방향이 불필요하게 확장되는 것을 막고, 작은 사업장 내부 도구로 바로 구현 가능한 기준을 제공하기 위해 작성합니다.

## 2. 제품 개요

### 제품명

Seolin SafeCheck

### 제품 유형

- 기사님용 모바일 웹앱/PWA
- 관리자용 PC 웹 대시보드
- Synology NAS 기반 백엔드 서버
- DB 중심 출결 관리 시스템
- 엑셀 가져오기/내보내기 지원 도구
- SMS API 연동 시스템

### 핵심 목표

등원 차량 기사님이 원생의 탑승 여부를 빠르고 정확하게 기록하고, 관리자가 날짜별 출결 기록을 확인하며, 탑승 처리된 원생의 학부모에게 문자 알림을 발송할 수 있는 운영 시스템을 구축합니다.

## 3. 문제 정의

현재 차량 출결 관리가 엑셀, 수기 명단, 전화, 구두 전달에 의존하면 다음 문제가 생길 수 있습니다.

1. 기사님이 운행 중 원생 탑승 여부를 빠르게 기록하기 어렵습니다.
2. 관리자가 날짜별, 호차별 기록을 체계적으로 확인하기 어렵습니다.
3. 학부모가 자녀의 탑승 여부를 즉시 알기 어렵습니다.
4. 엑셀 파일을 직접 수정하는 방식은 누락, 중복, 파일 손상 위험이 있습니다.
5. 기사님과 관리자 사이의 전달 지연으로 실제 상태와 기록이 달라질 수 있습니다.
6. 출결 기록이 누적될수록 이전 기록을 찾고 정리하기 어려워집니다.

## 4. 제품 목표

### 4.1 사용자 경험 목표

- 기사님이 한 원생의 상태를 빠르게 변경할 수 있어야 합니다.
- 기사님이 매번 로그인하지 않아도 되도록 로그인 상태를 유지합니다.
- 기사님 화면은 큰 버튼, 명확한 글자, 단순한 동작을 중심으로 구성합니다.
- 관리자는 엑셀과 유사한 표 형태로 원생과 시간표를 관리할 수 있어야 합니다.
- 학부모는 별도 앱 설치 없이 문자로 탑승 알림을 받을 수 있어야 합니다.

### 4.2 운영 목표

- 차량, 시간대, 원생, 탑승 장소, 연락처를 DB에서 중앙 관리합니다.
- 날짜별 출결 기록을 DB에 저장합니다.
- 출결 저장 이후 문자 발송 요청과 결과를 기록합니다.
- 엑셀은 원본 데이터가 아니라 가져오기/내보내기 수단으로만 사용합니다.
- 향후 SMS 제공업체가 바뀌어도 핵심 출결 저장 로직에 영향이 적도록 설계합니다.

## 5. 사용자 정의

### 5.1 기사님

기사님은 모바일 웹앱/PWA를 사용합니다.

주요 역할:

- 호차 계정으로 로그인
- 로그인 상태 유지
- 본인 호차의 시간대별 탑승 명단 확인
- 원생별 탑승 장소 확인
- 원생별 탑승 상태 변경
- 전체 저장
- 필요한 경우 임시 탑승 원생 추가

제한 사항:

- 정식 원생 정보 수정 불가
- 학부모 연락처 수정 불가
- 시간표 수정 불가
- SMS 설정 변경 불가
- 전체 관리자 데이터 접근 불가

### 5.2 관리자

관리자는 PC 웹 대시보드를 사용합니다.

주요 역할:

- 관리자 로그인
- 차량 관리
- 기사님 계정 관리
- 시간대 및 운행 일정 관리
- 원생 추가, 수정, 삭제
- 원생별 탑승 장소 관리
- 학부모 연락처 관리
- 출결 기록 조회
- 문자 발송 기록 조회
- 엑셀 가져오기/내보내기
- 문자 발송 옵션 관리

### 5.3 학부모

학부모는 MVP에서 별도 앱을 사용하지 않습니다.

주요 역할:

- 자녀 차량 탑승 문자 수신

기본 정책:

- `탑승` 상태는 문자 발송 대상
- `미탑승` 상태는 기본적으로 문자 발송하지 않음
- 관리자가 옵션을 켜면 `미탑승` 문자도 발송 가능

### 6.2 관리자용 기능

1. 관리자 로그인

5. 원생 목록 관리
6. 원생 추가
7. 원생 수정

9. 원생별 탑승 장소 관리
10. 학부모 연락처 관리
11. 날짜별 출결 기록 조회

13. 문자 발송 설정 관리
14. 엑셀 가져오기
15. 엑셀 내보내기

## 7. 제외 범위

MVP에서는 다음 기능을 구현하지 않습니다.

- 하원 차량 관리
- 귀가 차량 관리
- 학부모용 앱
- 실시간 GPS 위치 추적
- 차량 이동 경로 추적
- 결제 관리
- 태권도 띠 또는 수련 단계 관리
- 상담 기록 관리
- 출석 통계 리포트 고도화
- 카카오 알림톡 연동
- CloudKit Sharing
- Core Data migration
- Native iOS/Android 앱 구현
- 운영용 SMS 제공업체 설정 완료
- 미탑승 문자 기본 자동 발송

## 8. 주요 사용자 시나리오

### 8.1 기사님 로그인

1. 기사님이 모바일 웹앱에 접속합니다.
2. 호차 계정 ID와 비밀번호 또는 PIN을 입력합니다.
3. 서버가 계정 정보를 확인합니다.
4. 로그인에 성공하면 해당 계정과 연결된 차량 정보가 로드됩니다.
5. 로그인 세션이 유지되어 다음 접속부터는 반복 로그인을 줄입니다.
6. 계정이 비활성화되었거나 권한이 없으면 안내 메시지를 표시합니다.

### 8.2 기사님 출결 체크

1. 기사님이 앱을 실행합니다.
2. 오늘 날짜와 본인 호차 정보가 표시됩니다.
3. 기사님이 운행 시간대를 선택합니다.
4. 해당 시간대에 탑승 예정인 원생 목록이 표시됩니다.
5. 각 원생 카드에는 이름, 탑승 장소, 현재 상태가 표시됩니다.
6. 기사님이 원생 상태를 `탑승` 또는 `미탑승`으로 변경합니다.
7. 상태를 변경하지 않은 원생은 `미확인`으로 남습니다.
8. 기사님이 `전체 저장` 버튼을 누릅니다.
9. 저장 전 확인 모달이 표시됩니다.
10. 확인 후 서버에 출결 기록 저장 요청을 보냅니다.
11. 서버가 DB에 출결 기록을 저장합니다.
12. 저장 완료 후 `탑승` 원생의 학부모에게 문자 발송 요청을 생성합니다.

### 8.3 관리자 원생/일정 관리

1. 관리자가 PC 웹 대시보드에 로그인합니다.
2. 차량 또는 호차를 선택합니다.
3. 시간대와 요일을 선택합니다.
4. 해당 시간대의 원생 명단을 엑셀형 UI로 확인합니다.
5. 원생을 추가, 수정, 삭제 또는 비활성화합니다.
6. 원생별 탑승 장소와 학부모 연락처를 입력합니다.
7. 저장하면 DB에 반영됩니다.
8. 기사님 앱은 다음 목록 조회 시 변경된 정보를 표시합니다.

### 8.4 엑셀 내보내기

1. 관리자가 날짜 또는 기간을 선택합니다.
2. 전체 차량 또는 특정 차량을 선택합니다.
3. 출결 기록을 조회합니다.
4. 엑셀 다운로드 버튼을 누릅니다.
5. 서버가 DB 데이터를 기준으로 xlsx 파일을 생성합니다.
6. 관리자가 브라우저에서 파일을 다운로드합니다.

## 9. 출결 상태 정의

| 상태값 | 한국어 표시 | 설명 | 기본 문자 발송 |
| --- | --- | --- | --- |
| `unchecked` | 미확인 | 기사님이 아직 상태를 선택하지 않은 상태 | 발송 안 함 |
| `boarded` | 탑승 | 원생이 차량에 탑승한 상태 | 발송 |
| `not_boarded` | 미탑승 | 원생이 차량에 탑승하지 않은 상태 | 기본 발송 안 함 |

기본 상태는 `unchecked`입니다. `not_boarded` 문자 발송은 관리자 설정에 따라 추후 활성화할 수 있습니다.

## 10. SMS 발송 정책

### 10.1 발송 시점

문자는 기사님이 원생별 상태 버튼을 누를 때마다 즉시 발송하지 않습니다. 문자 발송 요청은 기사님이 `전체 저장` 버튼을 눌러 해당 시간대 출결을 확정한 뒤 생성합니다.

### 10.2 기본 발송 대상

- 상태가 `boarded`인 원생의 학부모 연락처

### 10.3 기본 미발송 대상

- `unchecked`
- `not_boarded`

단, 관리자 설정에서 `미탑승 문자 발송` 옵션을 켠 경우 `not_boarded` 상태도 문자 발송 대상이 될 수 있습니다.

### 10.4 문자 예시

```txt
[서린태권도]
김서린 학생이 15:20에 만촌역 앞에서 승차하였습니다.
```

### 10.5 발송 방식

- 외부 SMS API 연동을 전제로 합니다.
- 실제 SMS 제공업체, 발신번호 등록, 과금 방식은 추후 확정 필요입니다.
- 개발 환경에서는 실제 문자를 발송하지 않고 mock provider를 사용합니다.
- 출결 저장 실패와 문자 발송 실패는 분리해서 처리합니다.
- 문자 발송 성공/실패 결과는 `SmsLog`에 저장합니다.

## 11. 관리자 관리 정책

- 관리자는 차량, 기사님 계정, 시간대, 원생, 탑승 장소, 연락처를 관리합니다.
- 실제 원본 데이터는 DB에 저장합니다.
- 엑셀은 DB를 대체하지 않습니다.
- 엑셀은 다음 용도로만 사용합니다.
  - 기존 명단 가져오기
  - 출결 기록 내보내기
  - 백업용 다운로드
  - 오프라인 검토용 문서 생성
- 기사님 앱에는 관리자 전용 데이터 편집 기능을 노출하지 않습니다.

## 12. 비기능 요구사항

### 12.1 사용성

- 기사님 화면은 모바일 세로 화면에 최적화합니다.
- 버튼은 손가락으로 쉽게 누를 수 있는 크기로 제공합니다.
- 화면당 주요 동작은 1~2개로 제한합니다.
- 글씨 크기는 충분히 크게 설정합니다.
- 색상만으로 상태를 구분하지 않고 텍스트도 함께 제공합니다.
- 저장 중에는 중복 클릭을 막습니다.

### 12.2 안정성

- 저장 실패 시 기사님에게 명확한 안내를 표시합니다.
- 문자 발송 실패 시 출결 기록과 별도로 실패 기록을 남깁니다.
- 서버는 중복 저장 요청을 안전하게 처리해야 합니다.
- 네트워크 오류 발생 시 다시 시도할 수 있어야 합니다.
- DB 백업 방식은 운영 전 확정합니다.

### 12.3 보안

- 기사님은 본인 호차 데이터만 조회할 수 있습니다.
- 관리자는 전체 데이터를 관리할 수 있습니다.
- API 요청은 인증된 사용자만 호출할 수 있어야 합니다.
- 관리자 API는 관리자 권한을 확인해야 합니다.
- 비밀번호와 PIN은 해시 처리합니다.
- 외부 접속 시 HTTPS 사용을 전제로 합니다.

### 12.4 개인정보

- 원생 이름, 학부모 연락처, 탑승 장소, 출결 기록은 개인정보로 취급합니다.
- 기사님 화면에는 운행에 필요한 최소 정보만 표시합니다.
- 관리자 화면 외부로 학부모 연락처가 불필요하게 노출되지 않도록 합니다.
- 서버 로그에 개인정보를 과도하게 남기지 않습니다.
- SMS API Key 등 민감 정보는 환경변수로 관리합니다.

## 13. MVP 성공 기준

MVP는 다음 조건을 만족하면 성공으로 판단합니다.

- 기사님이 모바일에서 로그인할 수 있습니다.
- 기사님이 본인 호차의 시간대별 원생 목록을 확인할 수 있습니다.
- 기사님이 원생별 상태를 `탑승` 또는 `미탑승`으로 변경할 수 있습니다.
- 기사님이 `전체 저장`으로 해당 시간대 출결을 확정할 수 있습니다.
- 저장된 기록이 관리자 화면에서 날짜별로 조회됩니다.
- `탑승` 처리된 원생의 학부모에게 문자 발송 요청이 생성됩니다.
- 관리자가 원생, 차량, 시간대, 탑승 장소, 연락처를 수정할 수 있습니다.
- 관리자가 출결 기록을 엑셀로 다운로드할 수 있습니다.
- 엑셀이 원본 저장소가 아니라 DB 기반 import/export 도구로만 사용됩니다.
