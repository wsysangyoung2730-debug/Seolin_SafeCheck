# Seolin SafeCheck Code Rules

## 1. 문서 목적

이 문서는 Seolin SafeCheck 프로젝트에서 Codex 및 개발자가 코드를 작성할 때 반드시 따라야 할 규칙을 정의합니다.

목표는 다음과 같습니다.

1. 기능 범위가 불필요하게 확장되는 것을 방지합니다.
2. 기사님용 앱과 관리자용 앱의 책임을 분리합니다.
3. 데이터 저장 구조를 일관되게 유지합니다.
4. 문자 발송, 엑셀 처리, 인증 로직을 안전하게 관리합니다.
5. Codex가 기존 구조를 임의로 변경하지 않도록 합니다.

## 2. 기본 원칙

- 요청받은 범위 외의 파일은 수정하지 않습니다.
- 관련 없는 파일을 변경하지 않습니다.
- 기존 폴더 구조를 임의로 크게 변경하지 않습니다.
- 기능 추가 시 `README.md`, `docs/PRD.md`, `docs/ARCHITECTURE.md`와 충돌하지 않는지 확인합니다.
- MVP 범위를 벗어나는 기능은 임의로 구현하지 않습니다.
- 하원 차량, 귀가 차량, 학부모 앱, GPS 추적 기능은 MVP에서 구현하지 않습니다.
- CloudKit Sharing을 구현하지 않습니다.
- Core Data migration 로직을 추가하지 않습니다.
- 실제 데이터의 원본은 엑셀 파일이 아니라 DB입니다.
- 엑셀은 import/export 용도로만 사용합니다.
- 기사님 앱은 단순함을 최우선으로 합니다.
- 관리자 기능은 관리자 대시보드에만 둡니다.

## 3. 권장 프로젝트 구조

향후 구현 시 권장 구조는 다음과 같습니다.

```txt
seolin-safecheck/
├── README.md
├── docs/
│   ├── PRD.md
│   ├── ARCHITECTURE.md
│   └── CODE_RULES.md
├── src/
│   ├── app/
│   │   ├── driver/
│   │   ├── admin/
│   │   └── api/
│   ├── components/
│   ├── features/
│   ├── domains/
│   ├── lib/
│   ├── services/
│   ├── styles/
│   └── types/
└── public/
    └── assets/
```

이 문서 작업에서는 위 구조를 실제 구현하지 않습니다. 앱 구현 단계에서 기술 스택이 확정되면 필요한 폴더만 생성합니다.

## 4. 도메인 분리 규칙

기능은 다음 도메인 기준으로 분리합니다.

- `auth`
- `driver`
- `admin`
- `vehicle`
- `schedule`
- `student`
- `attendance`
- `sms`
- `excel`
- `settings`

각 도메인은 가능한 한 UI, API, 서비스 로직, 타입을 분리합니다. 기사님 기능에서 관리자 도메인의 수정 기능을 직접 호출하지 않도록 주의합니다.

## 5. 네이밍 규칙

### 5.1 파일명

- React Component: `PascalCase.tsx`
- Hook: `useSomething.ts`
- Service: `somethingService.ts`
- Type: `something.types.ts`
- Utility: `somethingUtils.ts`
- API helper: `somethingApi.ts`

예시:

```txt
DriverHomePage.tsx
AttendanceCard.tsx
useAttendanceSave.ts
attendanceService.ts
attendance.types.ts
smsService.ts
excelExportService.ts
```

### 5.2 변수명

- Boolean 값은 `is`, `has`, `can`, `should`로 시작합니다.
- ID 값은 대상 도메인을 명확히 포함합니다.

예시:

```ts
const isLoggedIn = true;
const hasUnsavedChanges = false;
const canSendSms = true;
const vehicleId = "vehicle_1";
const studentId = "student_1";
```

### 5.3 출결 상태값

출결 상태값은 다음 문자열만 사용합니다.

```ts
type AttendanceStatus = "unchecked" | "boarded" | "not_boarded";
```

UI 표시 문구는 별도 매핑으로 처리합니다.

```ts
const ATTENDANCE_STATUS_LABEL = {
  unchecked: "미확인",
  boarded: "탑승",
  not_boarded: "미탑승",
};
```

## 6. Driver UI 규칙

기사님용 UI는 다음 원칙을 따릅니다.

- 큰 버튼 사용
- 한 화면에 너무 많은 기능을 넣지 않음
- 상태 변경은 최대한 단순하게 구성
- 주요 버튼은 화면 하단 또는 명확한 위치에 배치
- 작은 아이콘만으로 기능을 표현하지 않음
- 색상과 텍스트를 함께 사용
- 저장 전 확인 모달을 반드시 제공
- 저장 중에는 중복 클릭을 막음
- 원생 상태 버튼은 선택된 버튼을 한 번 더 누르면 `미확인`으로 되돌리고, 반대 상태 버튼을 누르면 해당 상태로 전환합니다.

기사님용 화면에서 허용되는 주요 동작:

- 로그인
- 시간대 선택
- 원생 상태 변경
- 전체 저장
- 임시 탑승 원생 추가

기사님용 화면에서 금지되는 동작:

- 정식 원생 삭제
- 학부모 연락처 수정
- 호차 시간표 수정
- 차량 정보 수정
- SMS 설정 변경
- 관리자용 전체 출결 수정
- 전체 원생 목록 관리

## 7. Admin UI 규칙

관리자용 UI는 다음 원칙을 따릅니다.

- PC 화면 기준으로 설계
- 엑셀과 유사한 표 기반 관리 UI 제공
- 실제 저장은 DB에 수행
- 저장 버튼을 명확히 제공
- 삭제 기능은 확인 모달을 반드시 제공
- 원생 정보, 연락처, 탑승 장소, 시간표 수정 가능
- 변경 사항 저장 전/후 상태를 명확히 표시
- 대량 수정 기능은 검증과 확인 단계를 포함

권장 색상:

| Background | `#F7F8FC` |
| Text Primary | `#111827` |
| Text Secondary | `#6B7280` |

| Warning | `#F59E0B` |
| Danger | `#DC2626` |
| Border | `#E5E7EB` |

상태 색상:

- 미확인: 회색 계열
- 탑승: 초록 계열
- 미탑승: 빨강 또는 주황 계열

규칙:

- 색상만으로 상태를 판단하게 만들지 않습니다.
- 모든 상태에는 텍스트 라벨을 함께 표시합니다.
- 기사님용 버튼은 충분한 터치 영역을 확보합니다.
- 관리자 표는 읽기 쉬운 행 높이와 명확한 구분선을 사용합니다.

## 9. 인증 규칙

### 9.1 기사님 로그인

- 기사님은 호차 계정 ID와 비밀번호 또는 PIN으로 로그인합니다.
- 로그인 성공 후 자동 로그인 상태를 유지합니다.
- 자동 로그인 토큰은 안전하게 저장합니다.
- 기사님은 본인 호차 데이터만 접근할 수 있습니다.
- 기사님 계정이 비활성화되면 자동 로그인도 사용할 수 없어야 합니다.

### 9.2 관리자 로그인

- 관리자는 전체 데이터에 접근할 수 있습니다.
- 관리자 API는 관리자 권한이 있는 사용자만 호출할 수 있습니다.
- 기사님 권한으로 관리자 API를 호출할 수 없어야 합니다.
- 현재 개발용 admin 토큰은 로컬 개발 검증용이며 운영 인증으로 사용하면 안 됩니다.
- 민감한 관리자 작업은 가능하면 변경 이력을 남깁니다.

## 10. 데이터 저장 규칙

- 모든 원본 데이터는 DB에 저장합니다.
- 엑셀 파일을 원본 데이터 저장소로 사용하지 않습니다.
- 출결 기록은 날짜별로 저장합니다.
- 출결 기록은 원생, 호차, 시간대, 날짜, 상태를 포함해야 합니다.
- 문자 발송 결과는 출결 기록과 별도로 저장합니다.
- 삭제가 필요한 주요 데이터는 가능하면 soft delete를 우선 고려합니다.

예시:

```ts
interface SoftDeleteFields {
  isActive: boolean;
  deletedAt: string | null;
}
```

## 11. 출결 저장 규칙

- 기사님이 `전체 저장` 버튼을 누를 때만 서버에 출결 기록을 확정 저장합니다.
- 원생별 버튼 클릭 즉시 문자 발송을 하면 안 됩니다.
- 저장 전 확인 모달을 표시합니다.
- 저장 중 중복 클릭을 방지합니다.
- 저장 실패 시 사용자가 다시 시도할 수 있어야 합니다.

출결 저장 순서:

```txt
1. 클라이언트에서 상태 변경
2. 전체 저장 버튼 클릭
3. 저장 확인 모달 표시
4. 서버에 출결 저장 요청
5. DB에 출결 기록 저장
6. 탑승 상태 원생에 대한 SMS 발송 요청 생성
7. SMS 발송 결과 저장
8. 클라이언트에 저장 완료 응답
```

## 12. SMS 발송 규칙

### 12.1 기본 발송 조건

기본 문자 발송 조건은 다음과 같습니다.

```ts
attendance.status === "boarded";
```

### 12.2 기본 미발송 조건

다음 상태는 기본적으로 문자 발송하지 않습니다.

```ts
attendance.status === "unchecked";
attendance.status === "not_boarded";
```

단, 관리자 설정에서 미탑승 문자 발송 옵션이 활성화된 경우 `not_boarded` 상태도 발송 가능합니다.

### 12.3 문자 발송 모듈 분리

문자 발송 로직은 API handler 안에 직접 길게 작성하지 않습니다. 반드시 별도 서비스와 provider로 분리합니다.

예시:

```txt
services/sms/smsService.ts
services/sms/providers/mockSmsProvider.ts
services/sms/providers/realSmsProvider.ts
```

### 12.4 개발 환경

개발 환경에서는 실제 문자를 발송하지 않습니다. 개발 환경에서는 mock provider를 사용합니다.

```ts
if (process.env.NODE_ENV === "development") {
  useMockSmsProvider();
}
```

## 13. Excel 처리 규칙

- 엑셀 파일은 import/export 용도로만 사용합니다.
- 엑셀 업로드 시 데이터 검증을 반드시 수행합니다.
- 잘못된 연락처, 빈 이름, 중복 원생은 경고 처리합니다.
- 엑셀 다운로드는 DB 데이터를 기준으로 생성합니다.
- 기존 엑셀 파일의 셀 색상, 병합, 서식에 핵심 로직이 의존하면 안 됩니다.
- 가져오기 전 미리보기와 검증 결과 표시를 권장합니다.
- 가져오기 실패 시 어떤 행에 문제가 있는지 알 수 있게 안내합니다.

## 14. API 규칙

API는 다음 순서를 따릅니다.

1. 인증 확인
2. 권한 확인
3. 입력값 검증
4. 비즈니스 로직 실행
5. 에러 처리
6. 응답 형식 통일

응답 형식 예시:

```ts
interface ApiResponse<T> {
  success: boolean;
  data?: T;
  error?: {
    code: string;
    message: string;
  };
}
```

권한 규칙:

- `/api/driver/*`는 기사님 권한으로 접근합니다.
- `/api/admin/*`는 관리자만 접근합니다.
- 기사님 API는 로그인 계정에 매핑된 차량 기준으로만 데이터를 반환합니다.
- 관리자 read API에서도 학부모 연락처 등 민감 정보는 필요한 범위만 반환합니다.

## 15. 에러 처리 규칙

- 사용자에게 보여주는 에러 메시지는 명확하고 짧게 작성합니다.
- 기사님용 에러는 행동을 안내해야 합니다.
- 관리자용 에러는 수정해야 할 데이터를 알려야 합니다.
- 서버 로그에는 개발자가 확인할 수 있는 상세 정보를 남기되, 개인정보를 과도하게 노출하지 않습니다.

기사님용 에러 예시:

```txt
저장에 실패했습니다. 인터넷 연결을 확인한 뒤 다시 시도해주세요.
```

관리자용 에러 예시:

```txt
학부모 연락처 형식이 올바르지 않습니다.
```

## 16. 개인정보 규칙

다음 정보는 개인정보로 취급합니다.

- 원생 이름
- 학부모 이름
- 학부모 연락처
- 탑승 장소
- 출결 기록

규칙:

- 불필요한 로그 출력 금지
- API 응답에서 필요 이상의 개인정보 반환 금지
- 학부모 연락처는 관리자 화면에서만 노출
- 기사님 화면에는 운행에 필요한 최소 정보만 표시
- SMS API Key는 환경변수로 관리
- 엑셀 다운로드 파일 접근 권한을 제한
- 백업 파일 보관 위치와 접근 권한을 명확히 관리

## 17. Codex 작업 규칙

### 17.1 요청 범위 제한

좋은 요청 예시:

```txt
기사님용 시간대 선택 화면만 구현해줘.
수정 범위는 src/app/driver/home 이하로 제한해줘.
```

나쁜 요청 예시:

```txt
전체 앱 만들어줘.
알아서 구조 잡아줘.
```

### 17.2 파일 생성 규칙

Codex가 새 파일을 만들 경우 다음을 지켜야 합니다.

- 왜 필요한 파일인지 설명
- 기존 파일과 역할이 겹치지 않도록 구성
- 너무 큰 파일 하나에 모든 기능을 넣지 않음
- 관련 없는 파일을 함께 수정하지 않음

### 17.3 임의 구현 금지

Codex는 다음을 임의로 추가하면 안 됩니다.

- GPS 추적
- 학부모 앱
- 하원 차량 관리
- 귀가 차량 관리
- 결제 기능
- 상담 기능
- 태권도 띠 또는 수련 단계 관리
- CloudKit Sharing
- Core Data migration logic
- 외부 라이브러리 무단 추가

### 17.4 문서 작업 규칙

- 문서 작업 요청에서는 프론트엔드/백엔드 구현 코드를 만들지 않습니다.
- 문서 작업 요청에서는 빌드 도구를 설정하지 않습니다.
- 문서 작업 요청에서는 앱 런타임 동작을 변경하지 않습니다.

## 18. 테스트 규칙

MVP 기준으로 다음 테스트를 우선 작성합니다.

기사님 기능:

- 로그인 성공
- 로그인 유지
- 호차별 명단 조회
- 시간대별 명단 조회
- 상태 변경
- 전체 저장
- 저장 실패 처리
- 중복 저장 방지

관리자 기능:

- 원생 추가
- 원생 수정
- 원생 삭제 또는 비활성화
- 시간대 수정
- 호차별 명단 조회
- 출결 기록 조회
- 엑셀 다운로드

문자 기능:

- 탑승 상태 문자 발송 요청 생성
- 미확인 상태 문자 미발송
- 미탑승 상태 기본 문자 미발송

## 19. 커밋 규칙

커밋 메시지는 한국어로 작성하고, 태그는 소문자를 사용합니다.

권장 형식:

```txt
[feat] 기사님 출결 저장 기능 추가
[fix] 로그인 유지 오류 수정
[docs] PRD 문서 업데이트
[refactor] 문자 발송 서비스 분리
[style] 관리자 표 UI 간격 조정
[test] 출결 저장 테스트 추가
```

문서 초기 추가 커밋 예시:

```txt
[docs] Seolin SafeCheck 초기 문서 추가
```
